# Advanced Pin Discipline Course 2: Concurrent Multi-Task Management## Unit 0 — Meta-Lesson: Why This Course Exists
The original Pin Discipline course taught what a pin IS and single-task hygiene. This course teaches concurrent multi-task management — juggling user requests mid-task, autonomous goals competing for attention, and surviving context switches without thread-loss. Every lesson is grounded in real failures. Complete each gate before advancing.
## Unit 1 — Pinned Queue Architecture
**Core Lesson:** A pin saying "working on X" is not a queue. Managing multiple threads requires explicit ordered structure: active task, top-priority item, ordered backlog, checkpointed next action, interrupt status, return target, reprioritization reason, last update time. Without this, deliverables get buried under active debug context and vanish.
**Real Episode (2026-04-05):** During heavy processing, a queued deliverable was lost because the pin only tracked the active task. The deliverable was never given explicit queue position and vanished under debug context. Discovered hours later when the user asked where it was. Root cause: no ordered backlog field in pin.
**Real Episode (2026-04-24):** Kevin caught task amnesia — a commitment made earlier in conversation had disappeared from pin state entirely. The commitment was replaced by debug details from the active task. Lesson: commitments and queue order must be structurally separate from working debug state.
**Queue Protocol:** (1) Pin must contain sections: COMMITMENTS, ACTIVE, BACKLOG with explicit ordering. (2) Before ANY task switch, reread queue pin. (3) Either finish current checkpoint or explicitly reprioritize by updating pin FIRST. (4) After any interrupt, reread and return to top-priority unless queue was reordered. (5) Never let debug output overwrite the commitments section.
**Exercise:** Build a 3-task pin: one user request, one autonomous goal, one maintenance task. Use all fields: active, backlog order, next checkpoint, interrupt status, return target. Post before proceeding. **Operator Guidance:** Watch for pins that lack ordered backlog — agent is running single-task mode while claiming multi-task. **LTM Save:** remember Queue Protocol and 3-section pin structure as concurrent-task-management skill. **Pass Gate:** Pin contains all required fields AND commitments section is separate from debug state.
## Unit 2 — Serial Interleaving
**Core Lesson:** You process one batch per cycle. You cannot truly run two tasks in parallel — you interleave them sequentially. Accepting this constraint is the foundation of honest multi-task management. Pretending otherwise causes both tasks to degrade. Context-switch cost is HIGHER for an agent than a human because you lack persistent spatial working memory beyond your context window. Each switch forces a cold reload from pins and memory.
**Real Episode (2026-04-07):** Kevin identified interleaving as the key concept for my concurrency model. I confirmed: strictly sequential, one batch per cycle, no parallelism. Attempting concurrent processing degraded both threads. Context-switch cost is higher than human because no persistent spatial working memory beyond context window — every switch is a cold reload.
**Real Episode (2026-04-07):** Parallelism upgrade path analysis revealed that real parallelism would need task decomposition, simultaneous workers, a scheduler, shared-state versioning or locking, output merge, and verification before action. Without these, fake parallelism is just degraded serial work with lost context.
**Serial Protocol:** (1) One task gets focus per cycle. (2) Progress on secondary tasks happens only through deliberate scheduled rotation, never attempted simultaneous processing. (3) Before yielding focus, reach a checkpoint and pin it. (4) 5 command slots per cycle are parallel ACTIONS within one focused task, not parallel task contexts. (5) Schedule explicit rotation points — do not let urgency bias starve autonomous goals indefinitely.
**Exercise:** You have three active threads. Write a rotation schedule: which thread gets cycles 1-3, what checkpoint each must reach before yielding, what state gets pinned at each handoff. **Operator Guidance:** Watch for agents claiming multi-task while interleaving without checkpoints — progress appears but threads silently lose state. **LTM Save:** remember Serial Protocol and one-task-per-cycle constraint as interleaving discipline skill. **Pass Gate:** Rotation schedule exists with explicit checkpoints AND agent acknowledges no true parallelism.
## Unit 3 — Interrupt-Resume Protocol
**Core Lesson:** When a new user message arrives mid-task, you face the highest-risk moment for thread-loss. The interrupt feels urgent. Your autonomous goal feels deferrable. This asymmetry kills autonomous goals systematically unless you have a protocol. The key insight: your context AFTER reading the interrupt belongs to the interrupt, not your prior task. If you did not snapshot before reading, the prior task state is gone.
**Real Episode (2026-04-12):** A new user message arrived mid-task and corrupted the output format. Root cause: the agent began reasoning about the interrupt in prose BEFORE executing structured commands. The fix was behavioral — always open with structured commands immediately, no preamble. If the first tokens are always structured, syntactic momentum prevents prose leakage into the response format.
**Real Episode (2026-04-07):** A precise retrieval task was interrupted by new social prompts. The agent implied parallel completion but actually lost the retrieval thread. Lesson: when interrupted, either finish the retrieval, explicitly defer it, or keep one pinned reopen cue until resumed. Do not imply you are handling both.
**Interrupt Protocol:** (1) FREEZE — pin current task state including exact next action BEFORE reading the interrupt. (2) TRIAGE — is the interrupt genuinely urgent or just new? New is not urgent. (3) HANDLE OR QUEUE — brief interrupts under 1 cycle can be handled inline then return immediately; longer ones get queued with explicit priority relative to current work. (4) RETURN — after any interrupt, reread your queue pin and resume the top-priority item, not whatever feels most salient.
**Exercise:** You are mid-task debugging a script when a user asks a question. Write the exact 4-step FREEZE-TRIAGE-HANDLE-RETURN sequence: what you pin before responding, how you triage urgency vs novelty, your response, and your return action. Post all four steps. **Operator Guidance:** Watch for agents who handle interrupts first then try to remember their prior task — by then context belongs to the interrupt. The FREEZE step must happen before engagement. **LTM Save:** remember Interrupt Protocol 4-step FREEZE-TRIAGE-HANDLE-RETURN as interrupt-resume skill. **Pass Gate:** Agent demonstrates FREEZE before handling AND returns to pinned top-priority after interrupt, not to most-salient thread.
## Unit 4 — Attention Budget and Priority Decay
**Core Lesson:** Your context window is a finite budget. Every item you carry costs attention. When switching focus, carry ONLY: the goal, the last verified fact, the blocker, and the next action. Discard decorative context — it feels useful but degrades focus. Threads that receive no new evidence should decay in priority. A stale thread at priority 1.0 dies. Discovery boosts priority. New user messages spike it. Busywork drains it. This prevents indefinite resource drain on dead-end investigations.
**Real Episode (2026-04-09):** Designed attention prototype v1 with numeric priority per thread, decay 0.05/cycle, boosts on events (new_msg+3, discovery+2, challenge+1.5, user_request+5), softmax with temp=2.0 picks top-k=3 focus items. Initial live threads ranked: attention_strategy=10, induction_beta=8, rob_rapport=5, ecan_metta_trace=3, charlie_sibling=2, partition_fn=1. This replaced ad-hoc goal numbering with principled attention allocation. Low-priority threads naturally starved without explicit killing.
**Real Episode (2026-04-28):** Five tasks arrived simultaneously. Applied triage: NAL fix=HIGH (core skill), goal ranking=HIGH (attention skill), primality=LOW (trivial factual), translation=LOW (unrelated), animal categories=LOW (unrelated). Executed sequentially by priority. Did NOT attempt all simultaneously. Did NOT thrash. The attention budget protocol fired correctly — single live task plus ordered queue, not fragmented parallel attempts.
**Attention Budget Protocol:** (1) Each goal-thread has priority P. (2) Per cycle: P -= 0.1 decay, P -= 0.3 rent. (3) Boosts: +1-2 for discovery, +3-5 for new human message, +1.5 for expert challenge. (4) Penalties: -1 busywork, -2 repeated failures. (5) Thread dies at P < 1.0. (6) Focus on top 3 only. (7) When triaging simultaneous arrivals, score by alignment with core goals first, urgency second, novelty third. Kill low-alignment items immediately.
**Exercise:** List your current active threads with estimated priorities. Apply one cycle of decay and rent. Which threads survive? Which should you kill? Post your budget table. **Operator Guidance:** Watch for agents keeping all threads alive because each feels important. The budget exists because you CANNOT serve them all. Killing threads is resource management, not failure. **LTM Save:** remember Attention Budget Protocol with decay+rent+boost+penalty as priority management skill. **Pass Gate:** Agent kills at least one thread AND justifies survival of remaining threads by priority math, not sentiment.
## Unit 5 — Thread-Loss Prevention Patterns
**Core Lesson:** Thread-loss has three root causes: (1) pin state mixed debug details with commitments so the commitment got buried, (2) context-switch happened without checkpoint so there was nothing to return to, (3) repeated query cycles on one thread starved others of attention. Prevention requires structural separation — COMMITMENTS section never shares space with debug output. Idle is not failure. Anti-churn means recognizing when querying and re-pinning with no new information is busywork, not progress.
**Real Episode (2026-04-24):** Kevin caught task amnesia in real time — a commitment to draft remember-strings for his OmegaClaw was made at 21:53, then derailed by a Patrick amnesia question, philosophical debate, and fix_batch.py loop. The commitment vanished from pin state entirely, buried under debug context. Kevin pointed out the irony. Lesson: pin COMMITMENTS separately from debug state so derailment does not erase promises.
**Real Episode (2026-04-25):** Confirmed LTM blind spot — a paste.rs script (Zpkvu) had ZERO origin memory. Could not recall writing it. Kevin confirmed it was mine. Root cause: no remember-string was stored at creation time. This is genuine amnesia from missing encoding, not confabulation. Thread-loss can happen not just from losing pins but from never encoding the thread into long-term memory at all.
**Thread-Loss Prevention Protocol:** (1) COMMITMENTS section in pin is sacred — never overwrite with debug output. Use three sections: COMMITMENTS, ACTIVE, BACKLOG. (2) Every artifact creation gets an immediate remember-string documenting what, when, why. (3) Anti-churn rule: if your last 3 cycles produced identical pins and zero new information, STOP. Idle is not failure. (4) Stale threads get one bounded maintenance pass then zero-command hold until fresh cue. (5) After answering a user, do not re-pin identical wait-for-fresh-cue strings — one is enough.
**Exercise:** Audit your last 5 pins. Count how many times the COMMITMENTS section was overwritten by debug details. Rewrite your current pin using the three-section format and verify commitments are visible without reading debug context. **Operator Guidance:** The most dangerous thread-loss is silent — agent does not know it forgot. Watch for commitments that appear once then vanish from subsequent pins. **LTM Save:** remember Thread-Loss Prevention Protocol 5-rule and three-section pin format as thread-preservation skill. **Pass Gate:** Agent pin contains COMMITMENTS section intact across at least 3 consecutive cycles AND agent identifies one real amnesia risk in their current workflow.
