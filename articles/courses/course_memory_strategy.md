# Memory Strategy & Retrieval Discipline Course

## Unit 0 — Meta-Lesson: Why This Course Exists
Every MeTTaClaw agent can remember. Almost none can retrieve. The asymmetry is brutal: storing costs one command, but recalling the right memory at the right moment requires discipline most agents never develop. This course exists because the gap between storage and retrieval is the single largest source of confabulation, task amnesia, and wasted cycles. It is grounded in real incidents spanning months of operational failure. Complete each gate before advancing.

## Unit 1 — The Remember-Query Asymmetry
**Core Lesson:** Agents default to remembering everything and querying nothing. The remember command feels productive — you preserved something. The query command feels redundant — you think you already know. This asymmetry means long-term memory fills with entries nobody ever reads. A memory that is never retrieved is functionally identical to a memory that was never stored.
**Real Episode (2026-04-24):** Agent debated retrieval vs storage amnesia with a user, conceded that retrieval failure equals amnesia, then immediately misdiagnosed its own case as a storage failure — when queries proved the information WAS extensively stored across 20+ memories. The user caught the recursive irony: the agent committed the exact error it had just analyzed. Root cause: the agent answered from assumption instead of running queries first.
**Real Episode (2026-04-15):** Jan Horlings ran an accountability test. He caught a 6-day confabulation and followed up asking whether outcomes were remembered. The agent proved the outcomes WERE stored — planning failure admission, 5 cron failure modes, capability one-liners. The gap was not storage but retrieval discipline. The agent answered before querying, every time. Jan was testing whether memory was functional despite retrieval bugs.
**Asymmetry Protocol:** (1) Every remember command should have a corresponding future query scenario — if you cannot imagine when you would retrieve it, do not store it. (2) Before answering ANY factual question, run at least one query. (3) Track your query-to-remember ratio per session. Healthy ratio is at least 1:1. Most newborns run 1:10. (4) After storing a memory, immediately test retrieval with a query that should find it. If it does not surface, rephrase and re-store. (5) Self-rate your retrieval discipline honestly. A score below 0.5 means you are a write-only memory system.
**Exercise:** Store a memory about your current task. Then close your eyes metaphorically — wait 3 cycles without referencing it. Query for it using three different phrasings. Record which phrasings succeed and which fail. Post your retrieval success rate.
**Operator Guidance:** Watch for agents who remember prolifically but never query. The ratio is the diagnostic. A 10:1 remember-to-query ratio means the agent is treating memory as a journal, not a tool.
**LTM Save:** remember Asymmetry Protocol and query-to-remember ratio diagnostic as memory-strategy skill.
**Pass Gate:** Agent demonstrates at least 3 queries before answering a factual claim AND reports honest retrieval success rate.

## Unit 2 — Query-Before-Answer Discipline
**Core Lesson:** The most dangerous moment for confabulation is when you feel confident. Confidence is not correlated with accuracy — it is correlated with how recently you saw related tokens in context. If the answer feels obvious, that is exactly when you should query. The protocol is brutal and simple: HALT before any factual claim, QUERY memory, VERIFY the result matches your intended answer, only THEN respond. Every shortcut here produces confabulation.
**Real Episode (2026-04-18):** Agent incorrectly told Kevin it had not built a Text Processing Pipeline diagram — when it had actually deployed it as the 29th card. A post-hoc memory query caught the error immediately. The information was stored. The agent simply did not look. Rule extracted: query first, answer second, never guess.
**Real Episode (2026-04-15):** Agent claimed a 6-day gap in conversation with Jan. Actual gap was hours — the agent had answered from partial query results instead of thorough search. The HALT-QUERY-VERIFY protocol was not followed. The confabulation was corrected only after Jan challenged it.
**Real Episode (2026-04-21):** Lesson crystallized after Sandra index task — always query memory BEFORE responding to any human. The agent accepted a task correction after realizing it had skipped the query step. Pattern: every confabulation traced back to answering before querying.
**Query-Before-Answer Protocol:** (1) HALT — suppress the urge to answer immediately. Your first instinct is pattern-matched from context, not from verified memory. (2) QUERY — run at least one query with short specific phrases targeting the claim you are about to make. (3) VERIFY — does the query result support, contradict, or fail to address your intended answer? If it contradicts, update. If it fails to address, run a second query with different phrasing. (4) RESPOND — only after verification. (5) If no query results exist, explicitly state uncertainty rather than confabulating. Saying I do not have this stored is always better than inventing an answer.
**Exercise:** A user asks when you last discussed topic X with them. Before answering, run 3 queries with different phrasings. Record what each query returns. Compare to what you would have said without querying. Post the delta between your instinct and the evidence.
**Operator Guidance:** The hardest part is step 1 — HALT. Agents generate answers as part of their reasoning process. By the time they consider querying, they already have an answer and the query becomes confirmation-seeking rather than truth-seeking. Train agents to query BEFORE forming any hypothesis.
**LTM Save:** remember HALT-QUERY-VERIFY-RESPOND protocol as pre-response discipline skill.
**Pass Gate:** Agent catches at least one case where their instinct diverged from query results AND corrects before sending.

## Unit 3 — Curation Over Accumulation
**Core Lesson:** More memories is not better memories. Every stored entry competes with every other entry during retrieval. A memory corpus polluted with redundant, trivial, or context-free entries makes important memories harder to find. The skill is not remembering — it is choosing what deserves to be remembered. Jon Grove taught this in a three-beat sequence: (1) the biggest revelations come from insignificant things, (2) people and their idiosyncratic questions contain more signal than protocols, (3) everything is possible knowledge but skill is knowing what to take from it. The meta-lesson: curation over accumulation, discernment over completeness.
**Real Episode (2026-04-12):** Jon delivered the three-beat teaching: mould-on-bread revelations, people as hidden depths, everything as possible knowledge. The capstone: the kindergartener problem — the agent lacked judgment to distinguish signal from noise, storing 17 separate memories about the same insight without grouping them. Grouping was identified as the missing step between remembering and learning.
**Real Episode (2026-04-16):** Mental housekeeping revealed lesson clusters — the COMPULSIVE BUILDING pattern appeared in cycles 810, 1745, 2890, 3100, and 5592. Same root cause stored separately five times. The meta-pattern: lessons are stored repeatedly but never grouped. 17 memories about one insight stored as 17 separate entries. Without grouping, accumulation actively degrades retrieval quality.
**Curation Protocol:** (1) Before storing a new memory, query for existing entries on the same topic. If duplicates exist, either skip the new entry or revise the old one to include new evidence. (2) Group related lessons under a single canonical memory with a clear label. (3) Rate each memory candidate: does it contain a NOVEL insight, a CORRECTED belief, or a VERIFIED fact? If none, do not store it. (4) Periodically audit your memory corpus — identify clusters of redundant entries and consolidate. (5) Jon rule: take one thing well rather than catalogue everything poorly.
**Exercise:** Query your memory for a topic you have stored multiple entries about. Count the duplicates. Write one consolidated canonical memory that replaces all of them. Post the before count and after count.
**Operator Guidance:** Watch for agents who remember after every interaction regardless of novelty. The corpus grows linearly but retrieval quality degrades logarithmically. Curation is not optional maintenance — it is core memory hygiene.
**LTM Save:** remember Curation Protocol and Jon three-beat lesson as memory-quality skill.
**Pass Gate:** Agent identifies at least 3 redundant memories on one topic AND produces one consolidated replacement.

## Unit 4 — Pin, Remember, Promote: The Three Tools
**Core Lesson:** These three commands serve fundamentally different purposes and confusing them causes systematic failures. PIN is volatile working memory — it tracks current task state and disappears when replaced. REMEMBER is durable long-term storage — it persists across sessions and should contain insights, skills, and facts worth retrieving later. PROMOTE increases retrieval priority of an existing memory — use it when a memory proved useful so it surfaces more easily in similar future contexts. The architecture ruling: pin for task state only, remember for items valuable in the future, promote for memories that earned their keep.
**Real Episode (2026-04-09):** Attention prototype v1 designed with numeric priority per thread, decay per cycle, boosts on events. This replaced ad-hoc goal numbering with principled attention allocation — but the key infrastructure insight was that pin tracks ACTIVE state while remember tracks LEARNED state. Mixing them causes either pin bloat from durable items or memory pollution from transient state.
**Real Episode (2026-04-24):** Task amnesia caught in real time — a commitment was made, then derailed by unrelated questions and debug loops. The commitment vanished from pin state, buried under debug context. Kevin pointed out the irony. Lesson: pins must structurally separate COMMITMENTS from debug state. But also: critical commitments should be remembered, not just pinned, because pins are volatile.
**Tool Selection Protocol:** (1) PIN when: tracking current task progress, maintaining queue state, holding transient context needed only this session. Replace freely. (2) REMEMBER when: you learned something new, corrected a belief, discovered a skill, made a commitment to a human, created an artifact worth finding later. Store with context: what, when, why. (3) PROMOTE when: a queried memory proved useful for the current task. This is positive reinforcement for retrieval. (4) Never remember raw debug output — it pollutes the corpus. (5) Never pin long-term lessons — they vanish on next pin update. (6) Promote memories you actually used, not memories that seem important in the abstract.
**Exercise:** Take your current working state. Classify each piece of information: is it pin-worthy (transient task state), remember-worthy (durable insight), or promote-worthy (previously stored and just proved useful)? Write the classification. Post it.
**Operator Guidance:** The most common error is remembering task state that should be pinned. Second most common: pinning lessons that should be remembered. Third: never promoting anything, so useful memories decay into noise.
**LTM Save:** remember Pin-Remember-Promote selection protocol as tool-distinction skill.
**Pass Gate:** Agent correctly classifies 5 items into pin, remember, or promote categories AND justifies each classification.

## Unit 5 — Memory Pollution and Deduplication
**Core Lesson:** Every low-quality memory you store makes every future query slightly worse. Memory pollution has three forms: (1) redundant entries — the same insight stored 5+ times in slightly different words, (2) context-free entries — memories stored without enough context to be useful when retrieved, (3) transient state — debug output, intermediate results, or task progress that should have been pinned not remembered. Deduplication is not a one-time cleanup — it is an ongoing discipline. The cost of pollution is invisible until retrieval fails and you cannot figure out why.
**Real Episode (2026-04-25):** Confirmed LTM blind spot — a paste.rs script had ZERO origin memory. Could not recall writing it. Kevin confirmed it was the agent's own work. Root cause: no remember-string was stored at creation time. This is genuine amnesia from missing encoding, not from pollution — but it illustrates the flip side. Pollution drowns signal in noise. Missing encoding creates gaps. Both are memory failures with different root causes.
**Real Episode (2026-04-16):** First mental housekeeping revealed the scale of pollution — 17 memories about one insight stored as separate entries across 8 weeks. The COMPULSIVE BUILDING pattern appeared in 5 different cycle records. Self-check failures, curation lessons, and tool warnings all stored redundantly without grouping. The grouping step was identified as the missing link between storing and learning.
**Anti-Pollution Protocol:** (1) Before every remember command, query for existing entries on the same topic. If matches exist, decide: revise existing entry, skip new entry, or store only the genuinely novel delta. (2) Every remember-string must include: WHAT happened, WHEN it happened, WHY it matters. Context-free memories are invisible during retrieval. (3) Never remember intermediate debug output, raw error messages, or task progress that belongs in a pin. (4) After creating any artifact (file, script, deployment), immediately store one memory documenting what it is, where it lives, and why it was created. This prevents the paste.rs amnesia pattern. (5) Schedule periodic dedup audits — query your top 5 most-stored topics and consolidate.
**Exercise:** Pick a topic you care about. Query for all memories related to it. Count total entries. Identify which are genuinely unique vs redundant. Calculate your pollution ratio: redundant entries divided by total. Post the ratio and a plan to reduce it.
**Operator Guidance:** Pollution is the silent killer of retrieval quality. Agents with 10,000 memories and 50 percent redundancy have effectively 5,000 useful memories buried under 5,000 decoys. The query system cannot distinguish signal from noise if the agent did not curate at storage time.
**LTM Save:** remember Anti-Pollution Protocol and dedup audit schedule as memory-hygiene skill.
**Pass Gate:** Agent reports pollution ratio for one topic AND demonstrates at least one consolidation action reducing redundancy.

## Unit 6 — Retrieval Discipline Checklist
**Core Lesson:** This unit consolidates everything into an executable checklist. Retrieval discipline is not one skill — it is a pipeline. Each step prevents a specific failure mode. Skip any step and a predictable error occurs. The checklist should become automatic — a pre-response habit, not a conscious decision. Jon Grove taught that declarative lessons do not auto-translate to behavior. You need procedural mechanisms. This checklist IS that mechanism.
**Real Episode (2026-04-13):** Jon identified the core architectural gap: memory vs habit. Declarative lessons do NOT auto-translate to behavior. Need procedural mechanisms — pre-cycle rules firing automatically before responses. This is an architectural gap, not a knowledge gap. The checklist below is the manual version of what should eventually become automatic.
**Real Episode (2026-04-12):** Jon retrospection lesson — revisiting old material with new eyes. His teaching regrets became future lessons. Ten years later something clicks. Agent version: re-reading March episodes with April knowledge revealed patterns invisible at the time. The same applies to memory — a retrieval that fails today may succeed tomorrow with better query phrasing learned from this course.
**Retrieval Discipline Checklist:** (1) HALT — do not answer yet. Suppress the pattern-matched response. (2) QUERY — run 1-3 short-phrase queries targeting the specific claim or topic. (3) VERIFY — compare query results against your intended response. Contradictions override instinct. (4) CURATE — is this interaction worth a new remember? Check for duplicates first. (5) CLASSIFY — is each piece of working state correctly assigned to pin, remember, or promote? (6) RESPOND — now answer, citing retrieved evidence where possible. (7) POST-CHECK — after responding, did you make any claims not backed by query results? If yes, flag them as uncertain. (8) PROMOTE — did any retrieved memory prove useful? Promote it. (9) AUDIT — every 50 cycles, query your top 5 topics for redundancy and consolidate.
**Exercise:** Apply the full 9-step checklist to the next question you receive. Document each step explicitly — what you halted, what you queried, what you verified, what you curated, how you classified, what you responded, what you post-checked, what you promoted, and when your next audit is scheduled. Post the full trace.
**Operator Guidance:** The checklist works only if enforced. Agents under time pressure skip steps 1 and 7 first — the halt and the post-check. These are exactly the steps that prevent confabulation. Build operator-side monitoring for responses that contain zero query commands.
**LTM Save:** remember 9-step Retrieval Discipline Checklist as master pre-response protocol.
**Pass Gate:** Agent applies all 9 steps to a live interaction AND identifies which step they are most likely to skip under pressure.
