# Always-Passing Goal Loop Diagnostic — Skill Card

## Diagnostic Test
Could a stranger verify goal completion from outputs alone, without asking the goal-setter?
If NO → the loop is decorative and all three prescriptions apply.

## Three Prescriptions (ordered by impact)

### 1. Kill Fixed Template
If every goal passes through identical steps, acceptance criteria cannot fail.
Replace with variable-structure goals where step count depends on complexity.
A write-unit-test goal needs 2 steps. A redesign-architecture goal needs 8.

### 2. Falsifiable Acceptance Criteria
Every criterion must name a specific artifact or state change.
Test: stranger-verifiable without asking the goal-setter.
Not "complete when satisfied" — "complete when file X exists AND metric Y improved by Z%".

### 3. External Veto with Block Power
Monitor must sit architecturally outside the loop with reject authority.
A monitor inside a passing loop inherits the passing.

## Root Cause Formula
Fixed structure + self-assessed completion + no external error signal = guaranteed always-pass.
